# バックエンド API エラーレスポンス標準化に関する引継ぎ資料

## 1. 概要
これまで、API通信時にバックエンドから返却されるエラーレスポンスの形式（フォーマット）やステータスコードが各ハンドラでバラバラになっており、フロントエンド側でエラー文を直接解析（文字列パース）する必要があるなど、画面へのエラー表示が不安定になる課題がありました。

今回の改修により、**すべてのエラーレスポンスが統一されたJSON構造** になりました。今後フロントエンド側でエラーハンドリングを安全かつ確実に行うためのガイドラインとして本資料を共有します。

---

## 2. 新しいエラーレスポンスの統一構造

すべての400〜500番台エラーは、原則として**必ず以下のJSON構造で返却**されます。

```json
{
  "error": {
    "code": "エラーコード（大文字の英字）",
    "message": "エラーの詳細な説明"
  }
}
```

フロントエンド（Axios / ky / fetch 等）でエラーをキャッチした際、以下のように安全にエラー種別を判定できます。
```typescript
try {
  await api.post('/years', data);
} catch (error) {
  const errCode = error.response?.data?.error?.code;

  if (errCode === 'CONFLICT') {
    alert("すでに登録されています");
  } else if (errCode === 'INVALID_REQUEST') {
    alert("入力内容に誤りがあります");
  } else {
    alert("通信エラーが発生しました");
  }
}
```

## 3. 標準化されたエラーコード一覧表

今回の改修で定義された5つの共通 `error.code` は以下の通りです。

| HTTP Status | `error.code` | 発生する主なケース | フロントエンド側の対応例 |
| :--- | :--- | :--- | :--- |
| **400 Bad Request** | `INVALID_REQUEST` | バリデーションエラー（必須カラム不足・不当なEnum値指定・マイナス金額の送信など） | 「入力内容が正しくありません」などの警告表示 |
| **401 Unauthorized** | `UNAUTHORIZED` | 未ログインでAPIを叩いた、期限切れのトークンを使用した場合 | ログイン画面への遷移（リダイレクト）処理 |
| **403 Forbidden** | `FORBIDDEN` | アクセス権限不足（管理者用APIに一般のSponsorshipMemberがアクセスしようとした場合など） | 「権限がありません」等のメッセージ表示 |
| **404 Not Found** | `NOT_FOUND` | 存在しないID等のURLパラメータが指定された場合 | 一覧画面へのリダイレクト、または「データが見つかりません」表示 |
| **409 Conflict** | `CONFLICT` | データの重複（すでに登録されている同一会社名、同一ユーザーなど UNIQUE 制約による登録エラー等） | 「この会社はすでに登録済です」等の具体的な警告表示 |
| **500 Internal Server Error** | `INTERNAL_ERROR` | DBの接続切れなどサーバー側の予期せぬ内部エラー発生時 | 「時間をおいて再度お試しください」等の汎用メッセージ表示 |

---

## 4. 今回の大きな変更ポイント（注意点）

### 1) Enum値の厳密なバリデーション追加 (`400 Bad Request`)
データベースのEnum列に対する値チェックをバックエンド側に強化実装しました。
`companyStatus`, `phase`, `progress`, `productionType`, `status` などの項目に対し、不正な文字列（想定されていないタイポなど）を送信した瞬間に `INVALID_REQUEST` が返ります。フロントエンドの型定義（実装時の送信データ）にずれがないか注意してください。

### 2) UNIQUE制約エラーの安全な変換 (`409 Conflict`)
以前は同じ会社名を登録（重複）しようとした際、`500 Internal Server Error` と生のエラー文面（`Duplicate entry...`）がクライアントにまで漏洩していました。
今後は MySQLの `1062` エラーがバックエンド内で自動でキャッチされ、フロントエンドには安全な `409 CONFLICT` として返却されます。

以上です。フロントエンドのAPI接続・通信エラー制御の実装にお役立てください！
