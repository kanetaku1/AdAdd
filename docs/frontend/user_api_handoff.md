# ユーザー管理 API 実装とフロントエンドへの引継ぎガイド

バックエンド側でのユーザー管理機能（一覧取得、新規作成、更新）の実装が完了し、完全にモック（ダミー）データを脱却して本番稼働（MySQL連携）ができる状態となりました。

本ドキュメントは、フロントエンド開発チームへ向けた、新設APIの仕様と繋ぎ込み依頼に関する引継ぎ資料です。

## 1. 実装された API エンドポイントの仕様
すべてのエンドポイントは、本番環境のセキュリティ要件に基づき **Admin権限 (`X-User-Roles: admin`)** が必須となるルーティング保護（`RequireRoles("admin")`）がかけられています。

| HTTPメソッド | エンドポイント | 機能 | 備考 |
|---|---|---|---|
| `GET` | `/users` | ユーザー一覧の取得 | モックデータの代わりにこれを利用してください。 |
| `POST` | `/users` | 新規ユーザーの登録 | Emailが重複している場合、自動で `409 Conflict` エラーを返却します。 |
| `PATCH` | `/users/:id` | ユーザー情報の更新 | `isActive` (有効/無効フラグ) の boolean (true/false) 送信による切り替えにも完全に対応しています。 |

## 2. フロントエンドにおける受入要件と対応のお願い
現在フロントエンドの画面は、APIに接続されず **モックデータ (`mockUsers`) に直接依存した仮組みのまま** になっています。
このたびバックエンド側のAPIが本番仕様として完成いたしましたので、以下の画面・機能において **モック依存を解消し、上記API (`isApiEnabled()`分岐経由等) と連携するよう** プログラムの改修をお願いいたします。

### 【対応が必要な対象画面・機能】
1. **Users画面（ユーザー管理画面）**
   - ユーザー一覧を `GET /users` から取得して表示するように改修。
   - 画面からの新規作成操作を `POST /users` に送信するよう配線を繋ぐ。
   - ユーザーのステータス変更等の保存操作を `PATCH /users/:id` に送信するよう繋ぐ。
   - `409 Conflict` (Email重複時のエラー) 等のハンドリング表示。

2. **CompanyAssignment (担当者選択) のプルダウン/選択肢**
   - 担当者を割り当てる際の一覧選択肢を、ローカルの `mockUsers` ではなく `GET /users` から取得した MySQL 上の実在ユーザー（Member/Admin等）のみから表示・選択できるよう、`sponsorship.ts` 等の繋ぎ変えを実施。

## 3. その他の本番対応事項について
- *重複エラーハンドリング:* `POST` / `PATCH` 通信時、JSONボディに含まれる `email` が既にデータベース上に存在する場合、APIは `{ "error": "email already exists" }` というレスポンスと共に `HTTP 409 Conflict` を返します。フロント側でも「すでに使われているメールアドレスです」といったエラー表示への組み込みをお願いします。
- *テストのカバレッジ:* バックエンド側の Repository, Service, Handler の各階層に対して、バリデーションと動作を担保するテストを実装済みです。
